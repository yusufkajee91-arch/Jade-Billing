## 1. Executive Summary

This PRD defines the V2 feature set for dcco-billing, the legal billing platform for Dolata & Co Attorneys. The platform is being enhanced to become a subscription SaaS product marketed to both advocates and attorneys in South Africa.

The current system covers core billing, trust/business accounting, invoicing, FICA compliance, and reporting. V2 addresses gaps in practice management workflow (letters of engagement, client leads, task allocation), billing UX (unbilled fees page, timesheet enhancements), collections management, granular permissions, and a modernized diary/calendar experience.

This document is based on a gap analysis comparing Jessica's requirements against the current codebase (commit 50466aa on main).

## 2. Goals & Success Metrics

### Goals
1. **Complete the billing workflow** - Add unbilled fees page, pro forma invoicing, and fee schedule linking so the full capture-to-invoice pipeline is seamless.
2. **Add compliance modules** - Letters of Engagement and client-facing FICA portal to ensure firms can onboard clients properly.
3. **Improve daily UX for fee earners** - Timesheet stats, calendar-style diary, task allocation, and quick-add actions reduce friction.
4. **Enable multi-tenant readiness** - Zone-based permissions allow the system to serve firms of varying sizes with proper access control.
5. **Enhance collections** - Action tracking and interest calculation help firms recover outstanding fees.

**Problem:** There is no way to create, send, or track Letters of Engagement. Firms cannot ensure clients have formally engaged before billing.

**Requirements:**

| ID | Requirement | Priority |
|----|-------------|----------|
| LOE-1 | Create LOE from template linked to client and matter | Must Have |
| LOE-2 | Include fee schedule terms within LOE | Must Have |
| LOE-3 | Send LOE to client email for signing | Must Have |
| LOE-4 | Track LOE status (draft, sent, signed, expired) | Must Have |
| LOE-5 | Store signed LOE document against client record | Must Have |
| LOE-6 | E-signature integration (Adobe Sign or equivalent) | Should Have |
| LOE-7 | List all clients with LOE status, filterable | Must Have |
| LOE-8 | View agreed fee terms from LOE | Must Have |

**Data Model:**
```
LetterOfEngagement
  - id, clientId, matterId
  - templateContent, finalContent
  - status (draft, sent, signed, expired)
  - sentAt, signedAt, expiresAt

  - feeScheduleSnapshot (JSON of agreed rates)
  - createdBy, createdAt, updatedAt
```

**UI:** New `/compliance/loe` page with list view and create/edit form. Status badges. Link from client detail page.

## 5. Technical Architecture Notes

> **Reference:** All changes must comply with the layer hierarchy and dependency rules in [docs/dataflow.md](../../docs/dataflow.md). The key rule: Pages → Components → UI → Lib → Data. No upward imports. All DB access is server-side only.

### New Database Models Required

Adding models to `prisma/schema.prisma` triggers the schema change protocol:
1. Run `prisma generate`
2. Bump `PRISMA_SCHEMA_VERSION` in `lib/prisma.ts`
3. Check every API route that queries the changed model
4. Check components that display that model's data

| Model | Purpose | Consumed by |
|-------|---------|-------------|
| `LetterOfEngagement` | LOE tracking | api/loe/*, compliance/loe page, practice-overview, client detail |
| `ClientLead` | Lead/enquiry tracking | api/client-leads/*, clients/leads page, app-layout (quick-add) |
| `CollectionAction` | Collection workflow steps | api/collection-actions/*, collections page (debtors-view) |
| `FicaUploadToken` | Public FICA portal tokens | api/fica-portal/*, (public)/fica-upload page |
| `PermissionZone` + `UserZone` | Zone-based permissions | lib/auth.ts, all 81 consuming files, settings/users page |
| `ReceivedInvoice` | Accounts payable | api/received-invoices/*, business-account-view, diary (auto-entry) |
| `CollectionInterestConfig` | Per-client / firm-wide interest rates | api/collection-actions/*, collections page |

### New Lib Files Required (src/lib/)

Per the dataflow, all shared business logic lives in `src/lib/`. The following new files are needed:

| File | Exports | Consumed by |
|------|---------|-------------|
| `lib/permissions.ts` | `hasZone()`, `hasAnyZone()`, zone constants | All API routes (replaces role checks in lib/auth.ts session), sidebar (nav visibility), settings/users page |

### API Endpoints Required

All new API routes follow the standard pattern: import `auth.ts` + `prisma.ts`. New routes must use zone-based checks via `lib/permissions.ts` (not legacy role checks).

| Endpoint | Zone Required | Notes |
|----------|--------------|-------|
| `GET\|POST /api/loe` | Raise Invoices or Masterfile Manager | LOE CRUD |
| `GET\|PUT /api/loe/[id]` | Raise Invoices or Masterfile Manager | Individual LOE |
| `POST /api/loe/[id]/send` | Raise Invoices | Send LOE for signing |
| `GET\|POST /api/client-leads` | Masterfile Manager | Lead CRUD |
| `POST /api/client-leads/[id]/convert` | Masterfile Manager | Convert lead to client |
| `GET\|POST /api/collection-actions` | Collections | Collection action CRUD |
| `GET\|POST /api/fica-portal/[token]` | **No auth** (public) | Public FICA upload — validates token, not session |
| `POST /api/fica-portal/generate-link` | Masterfile Manager | Generate FICA upload link |
| `GET\|POST /api/received-invoices` | Bookkeeper | Accounts payable CRUD |
| `GET /api/trust-entries/balances-by-client` | Bookkeeper or Senior Bookkeeper | Trust balance summary |
| `GET /api/dashboard/upcoming` | General Access | "Coming Up" diary entries for dashboard |

### Migration Strategy for Permissions

1. Update `src/types/next-auth.d.ts` — add `zones: string[]` to the session user type
2. Create `permission_zones` and `user_zones` tables in `prisma/schema.prisma`
3. Run `prisma generate` + bump `PRISMA_SCHEMA_VERSION` in `lib/prisma.ts`
4. Seed 16 zones
5. Migrate existing users: map `admin` -> all zones, `fee_earner` -> General Access + Fee Earner + Fee Earner Reports, `assistant` -> General Access
6. Update `lib/auth.ts` JWT callback to embed zones in session — **this ripples to 81 consuming files**
7. Create `lib/permissions.ts` with `hasZone()` / `hasAnyZone()` helpers
8. Migrate API routes incrementally (can keep old role checks as fallback during transition)
9. Update `layout/sidebar.tsx` to show/hide nav items based on zones
10. Update UI to show/hide actions based on zones
11. Remove old `UserRole` enum once all routes are migrated
